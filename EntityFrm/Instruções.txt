Rodar projetos
* Rodar pelo vscode
- Dar get e atentar para não dar get com checkout
- Primeira vez que rodar:
File open folder, seleciona o projeto client do academico
ctrl + apostrofe abre o terminal
npm install, se precisar baixar algum pacote
.\CreateLinksScss.ps1, se der erro de font, ou arquivo específico
- npm run professor ou adm (ou pelo vs code ou pelo task runner, rodar aplicação/subir servidor)
- Rodar script .\AngularCheckout.ps1 pra conseguir comitar (se precisar dar get, dá undo na pasta raiz, no to all.. depois dá outro checkout)
- Segue exemplos ou desenvolve o que precisar
- Para dar checkin no projeto angular (até usar o git):
. Se criou algo novo tem que rodar .\AngularAdd.ps1
. Pending changes
. Undo na pasta raiz
. Ordena por change, e marca só edit (desmarcar arquivos adicionados, senão eles são apagados)
. Undo -> No to all, assim fica só a tela que foi alterada

* Rodar pelo visual studio
- Task runner explorer
- Seleciona SMC.Academico.Client -> Custom -> adm (dois cliques)
- Enquanto isso ele está rodando: npm run administrativo (tenta adm se der algo diferente)

* Se der erro de recursos, tenta dar um get no framework e rebuilda as solutions, e roda o power shell como administrador, e roda o .\CreateLinksScss.ps1 (Erro Recursos Angular)

Assim o servidor fica habilitado, senão o servidor cai, e não dá para logar na tela, ou vai pelo vs code ou vai pelo task runner explorer
Quando publicar, se a alteração não pegar, dá um ctrl f5, porque pode ficar cache do angular e não pegar a nova versão

Instruções
- Pai pro filho - bind
- Filho pro pai - evento
- Data-service - fica o retorno dos servicos para ficar global, se não tem que passar por estado
service - ficam as chamadas dos serviços, e os retornos ficam armazenados como um buffer no data-service
- O serviço simula o domainservice (se é só um lugar que vai chamar), mas também pode chamar uma controller + serviço + domainservice (para mais telas compartilharem, simula um api porque na controller tem autenticação e filtro de dados)
- Modulo é um agrupamento de componente, são os namespaces, e aí tem que registrar o módulo do componente, o módulo da área e o módulo da aplicação
- Nas views models não tem anotattion, porque montamos o html completo
- Subscribe - escutar os elementos para monitorar quais foram alterados (onChange), e fazer os dependencys, faz um valueChange com o campo que se alterado carrega o datasource de outro campo
todas as telas tem o carregarDados, que inicializa os objetos daquela tela no dataservice, e pode buscar esses valores no serviço 
- ModelSnapshot pega valor atual do modelo
- Debuga como um javascript, coloca um debugger no código e vai acompanhando pelo navegador
- Moment é usado para trabalhar com datas, valores.. se vai passar o valor pra outro método sempre dá um clone
- Controller = componente
- Filter = where
- Map = select
- FlatMap = selectmany
- Find = firstordefault
- Some = any
- Includes = contains
- Tentar sempre usar === ou !== que compara o valor e o tipo, se tiver certeza do tipo usa os ===, se não usa ==, se não tiver certeza de quais são os tipos o javascript tenta fazer a conversão para comparar
